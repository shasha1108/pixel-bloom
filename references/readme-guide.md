# README 写作指南

> 为 Claude Code Skill 项目生成结构化、有吸引力的 README。中文为主，英文为辅。

---

## 核心原则

1. **中文优先** — 标题、段落、叙述用中文；技术术语保留英文（p5.js、FSM、Perlin noise 等）；关键概念附英文对照
2. **前 10 秒定生死** — 用户扫一眼就要知道：这是做什么的、怎么用、为什么选它
3. **Show, don't tell** — 每个功能声明都要有具体例子或图示
4. **可扫读 > 面面俱到** — 用表格和列表，不用文字墙
5. **情感完整** — 技术文档也可以有温度，尤其是创作类工具

---

## 15 段标准结构

| # | 段落 | 必要性 | 说明 |
|---|------|--------|------|
| 1 | **Hero 区** | 必须 | 居中标题 + 一行定位语 + badges（stars/license/技术栈） + 快速导航链接 |
| 2 | **目录** | 必须 | `<details open>` 可折叠目录，锚点链接到各段 |
| 3 | **概述** | 必须 | 2-3 句说清：是什么、解决什么、与替代品的区别。紧接着用表格展示核心能力 |
| 4 | **产出展示** | 强烈建议 | ASCII 架构图 / 截图 / demo GIF——让用户看见"你能得到什么" |
| 5 | **快速开始** | 必须 | 3 步以内，可复制粘贴的命令。给出多个使用示例 |
| 6 | **模式/变体说明** | 按需 | 如果有多种模式（如场景模式），用对比表说明差异和适用场景 |
| 7 | **能力矩阵** | 必须 | emoji 前缀 + 表格，每个功能一行：名称、说明。用中文描述，技术术语保留英文 |
| 8 | **独特理念** | 按需 | 如果项目有独特的设计哲学或情感锚点，单独成段——这是区分度 |
| 9 | **工作原理** | 建议 | 流程图 + 可折叠详情 + 架构约束说明 |
| 10 | **质量门禁** | 按需 | 如果有测试/校验体系，用表格展示各层把关 |
| 11 | **技术栈** | 必须 | 居中表格：层级、技术、用途 |
| 12 | **项目结构** | 必须 | 目录树，每个文件一行注释说明用途 |
| 13 | **配置/参考** | 按需 | 色板、配置项、环境变量表 |
| 14 | **相关项目** | 建议 | 链接到同作者/同生态的其他项目 |
| 15 | **贡献 + 许可** | 必须 | Fork→Branch→PR 流程 + License 声明 + 收尾语 |

## 每段的写作规范

### Hero 区
```markdown
<div align="center">
<h1>ProjectName · 中文名</h1>
<p><em>一句话定位语，中文为主。</em></p>
<!-- badges row -->
<!-- 快速导航：快速开始 · 生成什么 · 能力矩阵 · 工作原理 · 项目结构 -->
</div>
```

### 目录
- 用 `<details open>` 默认展开
- 锚点链接用中文（`#-快速开始`）
- 控制在 15 个条目以内

### 概述
- 第一句直接说这是什么类型的工具
- 紧接一个能力表格（5-6 行），每行有 emoji + 技术名 + 中文说明
- 用 blockquote 强调最关键的差异化信息

### 快速开始
- 前提条件 1-2 行
- 主力示例用中文 prompt，再附英文版本
- 额外给 3-4 个灵感 prompt
- 用 blockquote 给出使用技巧

### 能力矩阵
```markdown
| | 能力 | 说明 |
|---|------|------|
| 🧬 | **中文能力名（English）** | 中文说明，技术术语保留英文 |
```

### 概念种子 / 设计哲学（如有）
- 引子用 blockquote 解释为什么有这个段落
- 表格四列：场景、表面需求、概念种子（加粗中文情感句）、北极星命名
- 结尾一段解释"这不是营销文案，而是决策锚点"

### 工作原理
- 用 ASCII 流程图展示管线
- 详情用 `<details>` 折叠
- 架构约束用代码块展示层级关系
- 如果用到了独特的设计法则，单独列表（法则 + 原则 + 实现）

### 质量门禁
- 表格：门禁名 + 工具/方式 + 检查内容
- 如果有独特的质量环节（如读者测试），用 blockquote 强调其独特性

### 技术栈
- 居中表格，层级列用 emoji
- 技术名称加粗
- 用途用简短中文

### 项目结构
- 完整的目录树
- 每个文件一行注释说明用途

### 贡献
- 数字列表：Fork → Branch → Commit → Push → PR
- 列出特别欢迎的贡献方向

## 语言对照规则

| 场景 | 语言选择 |
|------|----------|
| 段落标题 | 中文（## 🚀 快速开始） |
| 段落正文 | 中文 |
| 技术术语 | 英文（p5.js, FSM, Perlin noise, Ganzfeld） |
| 代码/命令 | 保持原样 |
| Badges | 英文（shields.io 标准） |
| 文件路径 | 保持原样 |
| 概念种子/情感内容 | 中文——这是项目的灵魂 |
| 收尾语 | 中文 |
| License 声明 | 英文（法律文本标准） |

---

## 反模式（不要做）

| ❌ 避免 | ✅ 改为 |
|---------|--------|
| 英文为主的 README（除非目标受众明确是英文社区） | 中文为主，英文为辅 |
| 开头就是安装命令 | 先说清这是什么、为什么值得用 |
| "这是一个……的工具" | "Skill 名称 是一个……，能……"——具体、主动 |
| 功能是大段文字 | 用表格，每行一个功能 |
| 只有一种安装方式 | 给多个使用示例 prompt |
| 没有架构/产出展示 | ASCII 图或截图展示"你能得到什么" |
| 隐藏边界和限制 | 诚实写出来（场景模式表、边界规则） |
| 忘记 License | 永远加上 |
| 纯 ASCII 像素画（渲染不一致） | 干净的标题 + badges 就足够好 |
| 快速开始只有 1 个例子 | 至少 3-4 个不同场景的 prompt |

## 检查清单

- [ ] Hero 区：标题 + 定位语 + badges + 导航链接，居中

- [ ] 概述第一句说清是什么
- [ ] 能力表格 5+ 行
- [ ] 快速开始 ≤ 3 步，有多个 prompt 示例
- [ ] 有产出展示（架构图/截图/ASCII）
- [ ] 技术栈表格完整
- [ ] 项目结构目录树完整，每个文件有注释
- [ ] 贡献流程清晰
- [ ] License 声明
- [ ] 中文为主，技术术语保留英文
- [ ] 没有纯 ASCII 像素画
- [ ] 回到顶部链接在每个大段末尾
